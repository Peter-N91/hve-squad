Consumer Guide

Bring your own PowerPoint template

The squad's PowerPoint Builder generates decks from your content. Give it your branded .pptx once and every deck inherits your slide masters, layouts, theme colors, and fonts. The branding rule ships with hve-squad and switches on by itself — a non-technical user never copies a file, edits a config, or restates the template on each request.

How branding flows in

When you ask the squad to build a deck, the Squad Coordinator dispatches the PowerPoint Builder, which runs the powerpoint skill's build_deck.py. Branding is applied through the --template flag (PowerShell -TemplatePath): the build inherits the template's slide masters, layouts, and theme (colors and fonts) and discards the template's own example slides, then adds only the slides defined in your content.

The template is everything. A polished deck comes from a well-built template far more than from the generator. Spend your effort on the master and layouts in PowerPoint, and the squad fills them in consistently.

Step 1 — Prepare a branded template

Open PowerPoint, build a 16:9 template, and save it as a .pptx. Make sure it has:

  • Slide Masters with your branded backgrounds, logo placement, and footer — set under View → Slide Master.
  • Named layouts the builder maps onto, at minimum Title Slide, Title and Content, and Section Header.
  • Theme colors and fonts defined in the theme (Slide Master → Colors / Fonts) so @theme_name references resolve to your brand palette.

You do not need to leave any example slides in the file — a --template build discards them and keeps only the masters, layouts, and theme.

Step 2 — Put it in a stable place

The template lives at a fixed path so the squad always knows where to find it. The convention: 

.github/brand/pptx-brand-template.pptx

You do not have to create that folder yourself. If you just hand the squad your .pptx (or point it at one), it offers to save a copy there for you — so a non-technical user never navigates the file tree.

Step 3 — It applies automatically

You do not need to copy, paste, or edit anything. hve-squad ships a branding instruction as part of the package, so on apm install it lands in your repo at .github/instructions/pptx-brand-template.instructions.md and switches on by itself. Because its applyTo matches **/.copilot-tracking/ppt/**, the PowerPoint Builder reads it every time it works on a deck.

The instruction tells the builder to use your template as --template for full rebuilds and --source for partial rebuilds. If no template is present yet, the builder does not quietly fall back to a plain deck — it asks you, in plain language, whether you have a branded .pptx, and offers to save the one you provide to .github/brand/pptx-brand-template.pptx for you. A non-technical user never has to touch a folder or a config file.

Full rebuild vs. partial rebuild. --template creates a new deck and keeps only masters, layouts, and theme — use it for new decks and full regenerations. For updating a few slides in an existing deck, the builder uses --source with --slides instead. Never combine the two: with both, --template wins and every slide not listed is lost.

Step 4 — Align content with the template (optional, technical)

You can skip this section — the squad handles it for you. If you want tighter control, two small settings keep slide content in sync with the brand theme. In content/global/style.yaml, declare the template and map layout names: 

template:
  path: ".github/brand/pptx-brand-template.pptx"
  preserve_dimensions: true
layouts:
  title:   "Title Slide"
  content: "Title and Content"
  section: "Section Header"

Then in each slide's content.yaml, reference colors with @theme_name rather than hardcoded hex values, so every slide adapts to whatever theme the template defines.

Step 5 — Always validate

The PowerPoint Builder's validation pass is what turns a correct deck into a polished one. It runs geometric checks (margins, overflow, spacing) and optional vision checks (overlaps, contrast, alignment, leftover placeholders), then loops back to fix issues. Let it iterate rather than accepting the first build — that loop is the difference between "it generated" and "it looks great."

Putting it together

  1. Build a branded .pptx with proper masters, named layouts, and theme.
  2. Hand it to the squad — or save it at .github/brand/pptx-brand-template.pptx; the squad will offer to place it for you.
  3. Ask the squad for a deck. The shipped instruction applies automatically — nothing to copy or configure.
  4. (Optional, technical) Set template and layouts in style.yaml and use @theme_name colors for tighter control.
  5. Let validation iterate to a clean result.

After that, a request as simple as /squad request="build a 6-slide overview deck for the booking service" produces an on-brand deck without any per-request template wiring.
